# ALai Studio v0.1

The first stable baseline: a working component library, Builder, Playground,
and static-site export flow, with 4 production-ready templates.

## What's included

- **Builder** (`/builder`) — compose multi-page projects from library
  components; edit props, style controls, and variants per section via the
  Inspector; drag-and-drop reordering; export the current project as a ZIP.
  Stable.
- **Inspector** — props/style/variant editing panel inside the Builder,
  driven entirely by each component's `manifest.ts`. Stable.
- **Playground** (`/playground/:componentId`) — isolated, local-only preview
  of a single component with live prop/style/variant editing, independent of
  any Builder project. Stable.
- **Export ZIP flow** — "Download ZIP" in the Builder generates a complete,
  standalone Vite + React project (components, pages, theme, assets,
  tooling) via the Export Engine. Stable, and now verified end-to-end
  (install + build + runtime) by `verify:templates` for every production
  template on every run.

4 full starter sites, browsable at `/templates`, each with a detail page at
`/templates/:id`:

| Template | Business | Pricing01 | Form01 |
|---|---|---|---|
| Agency (`Fieldnote Studio`) | Creative studio | ✅ | ✅ |
| Restaurant (`Luma Bistro`) | Neighborhood restaurant | — | ✅ |
| SaaS (`Northstar`) | Workflow software | ✅ | ✅ |
| Portfolio (`Mara Vale`) | Individual creator | — | ✅ |

All 4 have real, non-skeleton content (11–13 sections each) and none use
`Contact01` — `Form01` covers the inquiry/contact role everywhere it's used.

## Component library additions

- **Pricing01** — three-plan pricing section with an optional
  highlighted/recommended plan. Used in Agency and SaaS.
- **Form01** — contact/lead form with an optional side panel. **Static by
  default**: with no `formAction` configured, submitting shows a fixed demo
  notice and never navigates or sends data anywhere. Used in all 4 templates.

## Template thumbnails and detail pages

- Static thumbnail images (`public/template-thumbnails/*.png`) shown on
  `/templates` and `/templates/:id`, referenced via `Template.thumbnailPath`
  — presentational metadata only, never part of a `Project` and never
  exported.
- Each template has a dedicated detail page (`/templates/:id`) with a larger
  preview, description, tags, section list, and a clear "Use Template"
  primary CTA alongside a "Back to Templates" link.

## Export verification

`npm run verify:templates` drives the Builder, exports each of the 4
templates, and for each one:

- confirms the template card and Builder load correctly;
- exports and extracts the ZIP;
- confirms the expected component set;
- scans for template-metadata leakage;
- runs `npm install` and `npm run build` inside the extracted project;
- **serves the built project and runtime-checks it**: zero console errors,
  no failed image network requests, no broken `<img>` elements
  (`naturalWidth`/`naturalHeight`/`complete`).

The runtime image check is new in v0.1 — `npm run build` alone can't catch a
broken image path (it only proves the code compiles), so this closes that
gap for every template on every run.

## Current asset strategy

- **Templates are asset-free for now.** All 4 ship with `Project.assets: []`
  and every `*AssetId` prop empty — components fall back to inline SVG
  placeholders, never a broken `<img>`.
- **Thumbnails are metadata only.** `thumbnailPath` lives on `Template`, not
  `Project`, and is never exported.
- **No public `/assets/...` paths in template `Project.assets`.** Exported
  projects have no `public/` folder — a path that resolves inside this app
  would 404 in every standalone export.
- **Raster demo assets are deferred.** The Export Engine can only safely
  round-trip plain-text data URIs (inline SVG) into an exported project
  today; real photo support needs a dedicated, separately-reviewed Export
  Engine change. See [`docs/assets.md`](../assets.md) for the full rationale.

## Known non-blocking limitations

- **Bundle-size warning** — `npm run build`'s main JS chunk exceeds Vite's
  500 kB guidance threshold. Non-blocking; code-splitting may be worth
  revisiting later.
- **No full raster demo asset pipeline yet** — see "Current asset strategy"
  above.
- **No global `box-sizing: border-box` reset in exported projects** — the
  source app has one, generated exports don't. The currently-shipped
  components are fixed at the component level; adding the reset to the
  Export Engine's generated output would close this gap generically for
  future components, but is a separate, deliberate change.

## Recommended commands before future releases

```bash
npm run build             # confirms the source app compiles
npm run verify:templates  # confirms all 4 templates export, install, build, and run cleanly
```

See [`../export-verification.md`](../export-verification.md) for the full
list of what `verify:templates` checks, and the README's "When to run
`npm run verify:templates`" section for when it's required.
